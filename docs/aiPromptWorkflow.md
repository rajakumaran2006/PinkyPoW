## Problem Statement

Engineering students lose placements not from lack of ability, but from lack of calibrated practice. Existing platforms (LeetCode, Notion trackers) are static — they don't adapt to what you've already solved. PinkyPoW closes this gap with a stateful, adaptive AI engine that knows your exact history.

`generateJSON()` first checks `AI_PROVIDER` env var. If unset, it auto-detects by key presence: `GROQ_API_KEY` -> Groq; `GEMINI_API_KEY` -> Gemini. ### 2.5 Retrospective Highlights: Key AI Failures & Adaptations

To build a trusted, production-ready system, we honestly analyzed where AI models fell short during development and how we adapted our architecture to resolve these issues:

* **The Markdown-Wrapped JSON Incident:** When prompted to return raw JSON, models frequently wrapped output in markdown code blocks (` ```json `), causing `JSON.parse` to crash.
  * *Adaptation:* We moved away from simple negative prompts to **negative few-shot boundary examples** (explicitly showing the model what *not* to output) in [aiService.ts](file:///Users/raja/Desktop/ClientProjects/Hackathons/PinkyPoW/lib/aiService.ts).
* **Lenient Speech Grading:** Initial grading models defaulted to a generous grading bias from their training data (e.g., grading poor transcripts 8.5/10).
  * *Adaptation:* We replaced subjective ranges with **mechanical lookup rubrics** in [evaluateSpeech/route.ts](file:///Users/raja/Desktop/ClientProjects/Hackathons/PinkyPoW/app/api/ai/evaluateSpeech/route.ts) that enforce score caps unless concrete conditions (e.g., zero grammar errors, specific filler word count limits) are met.
* **Edge Middleware Limitations:** The AI assistant generated code that attempted to read `localStorage` inside Next.js edge middleware, which caused silent failures.
  * *Adaptation:* We migrated session states to **HTTP-only cookies** verified directly at the middleware layer in [middleware.ts](file:///Users/raja/Desktop/ClientProjects/Hackathons/PinkyPoW/middleware.ts).
* **Production Self-Healing Target:** As a lesson from duck-typed responses in minor routes, our next engineering sprint focuses on implementing **strict Zod schema validation** at all AI route boundaries to prevent state corruption.

## 3. AI Coding Assistant — Governance Rules & Configuration

This section documents the rules, constraints, and operating principles configured for the AI coding assistant (Antigravity) used throughout the PinkyPoW build. These are not arbitrary preferences — each rule exists because it solved a specific class of failure observed during development. The assistant operates under a planning-mode architecture with explicit approval gates.

### 3.1 Core Operating Principles

The AI assistant follows seven foundational rules that govern every interaction:

**Rule 1: Research Before Action.**
Before writing any code, the assistant must research the existing codebase — file structure, naming conventions, dependency versions, and established patterns. Code generated without context awareness introduces inconsistencies that compound across a hackathon timeline. This rule exists because early prompts (P-01 through P-02) produced CSS variables that conflicted with Tailwind's own variable namespace, requiring a full rewrite.

**Rule 2: Plan Before Execute.**
Any change that touches more than one file or introduces a new architectural pattern requires a written implementation plan before a single line of code is produced. The plan must identify: files to modify, dependencies between changes, and a verification method. This rule was introduced after the authentication system (P-03) was built with `localStorage`-based session tokens — a design that is architecturally impossible in Next.js edge middleware. A plan would have caught this in 30 seconds.

**Rule 3: Preserve Existing Conventions.**
All generated code must match the project's existing patterns: naming conventions, file organization, import style, error handling approach, and comment format. The assistant must never introduce a new pattern without explicit justification. This prevents the codebase from becoming a patchwork of inconsistent styles — a common failure mode when AI generates code in isolation from its surroundings.

**Rule 4: Verify Before Declaring Done.**
Every code change must be verified through at least one of: running the build, executing tests, or manual inspection of the output. "It should work" is never an acceptable verification. This rule was formalized after P-14 (DSA code evaluator) passed a visual review but failed `JSON.parse()` at runtime because the model wrapped its output in markdown fences that were invisible in the prompt design phase.

**Rule 5: Document Non-Obvious Decisions.**
Any design choice that a future developer (or the same developer at 3 AM) would question must be documented inline or in the commit message. This includes: why a specific model temperature was chosen, why a schema field uses a particular type, and why a prompt includes a counter-example. Self-documenting code is necessary but insufficient — the "why" matters more than the "what."

**Rule 6: Minimal Footprint Changes.**
Changes must be scoped to the smallest surface area that achieves the goal. If fixing a bug in `evaluateSpeech`, the assistant must not also refactor the import order in `aiService.ts`. Unrelated changes in the same commit create merge conflicts, obscure git blame, and make rollbacks dangerous. This rule was violated once during P-17/P-18 when a speech grading fix also reformatted the entire route file, making the actual change invisible in the diff.

**Rule 7: Never Guess Missing Requirements.**
If a user request is ambiguous — for example, "make the scores better" — the assistant must ask for clarification rather than interpreting the request. A wrong guess costs more time than a 30-second clarification. This rule was formalized after the assistant interpreted "improve the rubric" as "make scores higher" rather than "make scoring more discriminating," which produced the opposite of the intended outcome.

### 3.2 Prompt Discipline for the Coding Assistant

The AI coding assistant itself operates under strict prompt constraints that prevent waste and ensure quality:

**Constraint A: Single Responsibility Per Prompt.**
Each prompt to the assistant addresses exactly one concern. "Build the DSA page and also add the speech recorder" is two prompts, not one. Combining unrelated concerns in a single prompt increases the probability of partial failures and makes it impossible to rollback one change without losing the other.

**Constraint B: Context Injection Over Re-Explanation.**
Instead of re-explaining the project's architecture in every prompt, the assistant reads the actual source files. This means prompts reference specific file paths (`lib/aiService.ts`, `models/User.ts`) rather than describing what those files contain. The source code is the truth; any description of it is a potential source of drift.

**Constraint C: Explicit Output Boundaries.**
Every prompt that expects code output specifies exactly what the output should contain and what it should not. "Write a POST handler" is insufficient. "Write a POST handler that accepts `{action, clerkId}` in the body, validates types, calls `connectDB()`, and returns `NextResponse.json()` — do not add authentication middleware, do not modify the User model" is sufficient.

**Constraint D: Error Handling Is Not Optional.**
Every generated function must handle failure cases. The assistant is instructed to never produce a `try` block without a `catch` that: logs the error with context, returns an appropriate HTTP status code, and includes a human-readable error message. Silent failures are the most expensive kind of bug.

**Constraint E: No Speculative Features.**
The assistant must not add features, parameters, or code paths that were not explicitly requested. "While I'm here, I also added pagination support" is a violation. Unrequested features are untested features, and untested features are bugs in waiting. Every line of code must trace back to a specific requirement.

### 3.3 Planning Mode Architecture

The assistant operates in a three-phase workflow:

```
┌──────────────────────────────────────────────────────────────────┐
│  PHASE 1: RESEARCH                                               │
│  Read existing code. Identify patterns. Map dependencies.        │
│  No source code modifications allowed.                           │
├──────────────────────────────────────────────────────────────────┤
│  PHASE 2: IMPLEMENTATION PLAN                                    │
│  Write a detailed plan. Identify files to create/modify.         │
│  Document open questions. Flag breaking changes.                 │
│  ── USER APPROVAL GATE ──                                        │
│  Execution does not begin until the user explicitly approves.    │
├──────────────────────────────────────────────────────────────────┤
│  PHASE 3: EXECUTE & VERIFY                                       │
│  Write code. Run builds. Execute tests.                          │
│  Update task tracking. Create walkthrough on completion.         │
└──────────────────────────────────────────────────────────────────┘
```

This architecture exists because the most expensive AI coding failures are not wrong code — they are wrong direction. A five-minute plan review catches architectural mistakes that would take two hours to unwind after implementation.

### 3.4 Quality Gates: What Makes This Different From "Vibe Coding"

The term "vibe coding" describes a workflow where the developer accepts AI output without review, iterating by feel rather than by verification. PinkyPoW explicitly rejects this approach through three quality gates:

**Gate 1: Schema Validation on Every AI Route.**
Every API route that calls `generateJSON()` validates the response against an expected schema before persisting to the database. The model may rename fields, omit required properties, or wrap the response in an unexpected container. Schema validation catches these failures at the boundary, before they corrupt state.

**Gate 2: Prompt Counter-Examples.**
After the markdown-fencing incident (Section 6), every system prompt that requests structured output includes at least one counter-example showing what a bad response looks like. Telling the model "do not do X" is measurably weaker than showing it "here is what X looks like — never produce this."

**Gate 3: Mechanical Rubrics Over Subjective Instructions.**
After the speech grading inflation incident (Section 6), every scoring prompt uses a lookup-table rubric rather than a range instruction. "Grade on a scale of 1-10" produces unreliable scores because the model defaults to whatever scoring distribution exists in its training data. "Score >= 9 requires: zero grammar errors AND <= 2 filler words AND >= 45 seconds of speech" produces consistent, defensible scores.

## 4. Prompt Engineering Methodology — Taxonomy of Techniques

Seven techniques are applied systematically across all 30 prompts in the portfolio. Each technique exists because it solved a specific class of model failure.

### 4.1 System Instruction Isolation

System rules live in a dedicated message block (`role: "system"`); user data occupies a separate block (`role: "user"`). The two are never mixed. This separation exists because LLMs treat system instructions as higher-priority constraints. When rules and data coexist in the same block, the model occasionally treats data values as instructions — for example, interpreting a user's code comment `// TODO: add error handling` as an instruction to the model itself.

**Implementation:** Groq's chat completions API accepts an explicit `messages` array with distinct `system` and `user` roles. Gemini uses the `systemInstruction` parameter on the model configuration object. Both enforce the separation at the API level.

### 4.2 Contextual State Grounding

Before any LLM call is made, the API route queries MongoDB and injects the user's live state into the prompt. This includes `placementScore`, `dailyStreak`, `completedProblems[]`, `techStack[]`, and `currentDayIndex`. The model never operates in a vacuum — every response is calibrated against the user's actual progress.

**Why this matters:** A model that generates DSA problems without knowing what the user has already solved will produce duplicates. A model that generates a study roadmap without knowing the user's placement score will recommend the wrong difficulty level. State grounding is the single most important design decision that separates PinkyPoW from a stateless chat wrapper.

**Implementation in code:**

```typescript
// From app/api/ai/dsa/route.ts — state grounding before LLM call
const user = await getOrCreateUser(clerkId);
const dsaProgress = await DSA.findOne({ userId: user._id });
const dayPlan = dsaProgress.planRoadmap?.find(p => p.day === targetDay);

// Injected into prompt — model sees live DB state
const userPrompt = `Generate problems for Day ${targetDay}:
- Topic: ${dayPlan.topic}
- Level: ${dsaProgress.level}
- AVOID: ${JSON.stringify(dsaProgress.completedProblems)}
- Solved History: ${JSON.stringify(solvedHistory)}`;
```

### 4.3 Structured JSON Mode

Both providers enforce JSON output at the API level — not through prompt instruction alone. Groq uses `response_format: { type: "json_object" }`. Gemini uses `responseMimeType: "application/json"`. Both paths then call `JSON.parse()` with schema validation.

**Why API-level enforcement:** Prompt-level instructions like "return only JSON" have a measurable failure rate. The model occasionally prepends explanatory text ("Here is the JSON:") or appends commentary after the closing brace. API-level enforcement eliminates these failure modes entirely.

### 4.4 Few-Shot Injection

Sample input/output pairs in the system block define output boundaries. The model sees a concrete example of what a correct response looks like before processing the actual request. This technique is applied in every scoring prompt (DSA evaluation, speech grading, certification matching) to anchor the model's output format.

**Example from the DSA evaluator:**

```
Few-shot example (incorrect bubble sort):
USER CODE: for i in range(len(a)): for j in range(i): if a[j]>a[j+1]: a[j],a[j+1]=a[j+1],a[j]
EXPECTED OUTPUT: { "compiled": true, "passed": false, "cases": [...], "timeComplexity": "O(n²)" }
```

### 4.5 Chain-of-Thought Guardrails

System prompts include an explicit instruction to "check constraints before emitting JSON." This forces the model to internally verify its output against the schema before serializing. Without this instruction, the model occasionally produces structurally valid JSON that violates business constraints — for example, a `timeComplexity` value of "fast" instead of Big-O notation.

### 4.6 Negative Constraint Specification (Counter-Examples)

After the markdown-fencing incident (Section 6), every system prompt that expects structured output includes an explicit example of what a bad response looks like. The instruction "do not wrap output in markdown fences" was insufficient. The instruction "do NOT produce this: \`\`\`json { ... } \`\`\` — output ONLY the raw JSON object" is measurably more effective.

### 4.7 Mechanical Rubric Anchoring

For any prompt that requires scoring or grading, the system instruction includes a concrete lookup table mapping score values to observable criteria. This replaces subjective instructions like "grade on a scale of 1-10" with deterministic rules like "score >= 9 requires: advanced technical terminology present, zero grammar errors, <= 2 filler words, >= 45 seconds of speech."

**Why this works:** LLMs default to whatever scoring distribution exists in their training data. For most evaluation tasks, that distribution skews generous (most training examples of "grading" involve encouraging feedback). A mechanical rubric overrides this default and produces scores that correspond to actual performance.

## 5. State Grounding Deep-Dive — The Anti-Wrapper Architecture

This section exists to demonstrate, with concrete evidence, that PinkyPoW's AI is not a wrapper around an LLM API. A wrapper is a system where user input goes to the model and the model's output goes to the user, with no intermediate state, no context accumulation, and no feedback loop. PinkyPoW's architecture is the structural opposite.

### 5.1 The Feedback Loop

Every AI interaction in PinkyPoW mutates persistent state, and that mutated state is injected into the next AI interaction. This creates a closed feedback loop:

```
  ┌────────────────────────────────────────────────────────┐
  │                                                        │
  │    USER ACTION                                         │
  │        │                                               │
  │        ▼                                               │
  │    QUERY MONGODB  ───────►  BUILD PROMPT               │
  │        ▲                        │                      │
  │        │                        ▼                      │
  │    MUTATE STATE  ◄────────  LLM RESPONSE               │
  │                                                        │
  └────────────────────────────────────────────────────────┘
```

**Concrete example:** When a user solves a Medium DSA problem:

1. `placementScore` increases by 30 points in MongoDB.
2. The problem title is appended to `completedProblems[]`.
3. `dailyStreak` increments if all problems for the current day are solved.
4. The next time the user requests a new problem, the AI receives the updated `placementScore` (which may shift the difficulty band), the updated `completedProblems[]` (which prevents duplicates), and the updated `dailyStreak` (which influences motivational framing).

A wrapper would generate the same problem regardless of how many problems the user has already solved. PinkyPoW cannot, by construction, produce a duplicate problem — the `AVOID` list in the prompt grows with every solved problem.

### 5.2 Placement Score as a Difficulty Calibrator

The `placementScore` (0-1000, hard-capped) is not a cosmetic number. It is a functional parameter that changes the AI's behavior:

| Score Range | AI Behavior Change                                                           |
| ----------- | ---------------------------------------------------------------------------- |
| 0-349       | DSA problems are Easy difficulty. Roadmap focuses on fundamentals.           |
| 350-749     | DSA problems are Medium. Roadmap includes intermediate patterns.             |
| 750-1000    | DSA problems are Hard. Roadmap emphasizes advanced topics and system design. |

**Score mutation table:**

| Action                         | Score Delta | Rationale                                                  |
| ------------------------------ | ----------- | ---------------------------------------------------------- |
| Solve Easy DSA Problem         | +15         | Low reward to discourage grinding easy problems            |
| Solve Medium DSA Problem       | +30         | Balanced reward for standard difficulty                    |
| Solve Hard DSA Problem         | +55         | High reward; hard problems require disproportionate effort |
| Add a Certificate to Portfolio | +25         | Certification adds demonstrable credibility                |
| Unmark Easy DSA Problem        | -15         | Symmetric penalty prevents score gaming                    |
| Unmark Medium DSA Problem      | -30         | Same as above                                              |
| Unmark Hard DSA Problem        | -55         | Same as above                                              |
| Score floor                    | 100 (min)   | Prevents demoralization from negative spiral               |
| Score ceiling                  | 1000 (max)  | Prevents runaway inflation                                 |

This is the core mechanism that makes PinkyPoW adaptive. A wrapper has no score. A wrapper has no memory. PinkyPoW has both.

### 5.3 Duplicate Prevention via Solved History Injection

Every `generate-problem` call passes the full list of previously solved problem titles to the LLM as an explicit `AVOID` constraint:

```typescript
const userPrompt = `Generate problems for Day ${targetDay}:
- AVOID: ${JSON.stringify(dsaProgress.completedProblems || [])}
- Solved History: ${JSON.stringify(solvedHistory)}`;
```

As the user progresses through a 30-day plan, the `AVOID` list grows from 0 to potentially 60+ entries. The model must generate problems that are topically appropriate, difficulty-calibrated, and distinct from everything the user has already encountered. This constraint tightens progressively — the further the user advances, the harder the model must work to produce novel content.

### 5.4 Graceful Degradation with Local Fallbacks

The filter endpoints (`filterInternships`, `filterHackathons`, `filterCertifications`) implement a dual-layer strategy: AI-first with local keyword fallback. If the LLM call fails (API key missing, rate limit, network error), the route falls back to a local keyword-matching algorithm that uses simple string inclusion against the data fields.

```typescript
try {
  result = await generateJSON(systemPrompt, userPrompt);
} catch (aiError) {
  console.warn("AI filtering failed. Using fallback local search.", aiError);
  const keywords = query.toLowerCase().split(/\s+/).filter(w => w.length > 2);
  const matchingIds = items.filter(item => {
    const text = `${item.title} ${item.company}`.toLowerCase();
    return keywords.some(kw => text.includes(kw));
  }).map(item => item.id);
  result = { matchingIds };
}
```

This means the application never shows a blank error screen to the user. Even without any AI provider configured, the core filtering functionality works. A pure wrapper would crash.

## 6. Real Model Exchanges — Failure Transcripts and Corrections

### 6.1 DSA Code Evaluator — Markdown Fencing Incident (P-14)

The following is a lightly redacted transcript of the first integration test against `/api/ai/dsa`. The model's initial response failed schema validation; the corrected re-prompt and its output are shown.

**System prompt sent (abbreviated):**

```
You are a code evaluation engine. Given user code, language, problem ID, and test cases,
return ONLY valid JSON matching this exact schema — no markdown fences, no commentary:
{
  "compiled": boolean,
  "passed": boolean,
  "cases": [{ "input": string, "expected": string, "received": string, "passed": boolean }],
  "timeComplexity": string
}
Few-shot example (incorrect bubble sort):
USER CODE: for i in range(len(a)): for j in range(i): if a[j]>a[j+1]: a[j],a[j+1]=a[j+1],a[j]
EXPECTED OUTPUT: { "compiled": true, "passed": false, "cases": [...], "timeComplexity": "O(n²)" }
```

**First model response — FAILED validation:**

```json
```json
{
  "result": {
    "compiled": true,
    "passed": false,
    "cases": [ ... ],
    "complexity": "O(n log n)"
  }
}
```

```

> **Three bugs caught in one response:**
> 1. The model wrapped the JSON in markdown fences despite being told not to. `JSON.parse()` threw on the backtick characters.
> 2. The model used `"result"` as a root wrapper key — the schema expects the fields at the root level.
> 3. The model used `"complexity"` instead of `"timeComplexity"` — a silent rename that would have corrupted downstream analytics.
>
> **Root cause:** "Don't do X" is weaker than "here is what X looks like — never produce this."

**Fix applied to system prompt:**
```

CRITICAL: Do NOT wrap output in markdown fences (no ```json). Output ONLY the raw JSON object.
Root key must be the schema root — no wrapping object named "result" or any other key.
Field name is exactly "timeComplexity" — not "complexity".

```

**Second model response — PASSED validation:**
```json
{
  "compiled": true,
  "passed": false,
  "cases": [
    { "input": "[5,3,1,4,2]", "expected": "[1,2,3,4,5]", "received": "[1,2,3,5,4]", "passed": false }
  ],
  "timeComplexity": "O(n²)"
}
```

This transcript is the reason the system prompt in `lib/aiService.ts` defaults to an explicit anti-example alongside the schema definition.

### 6.2 Speech Grading Inflation Incident (P-17)

**Problem:** The speech evaluator used a system prompt that said "grade on a scale of 1-10." With no rubric, the model applied lenient grading. During internal testing, the grader awarded 8.5 to a transcript that was literally "um, yeah, I think it works."

**Root cause:** LLMs default to whatever scoring distribution exists in their training data. For evaluation/grading tasks, that distribution skews generous — most training examples of "feedback" are encouraging rather than critical.

**Fix:** P-18 adds a mechanical rubric:

- Score >= 9: advanced technical terminology, zero grammar errors, <= 2 filler words, >= 45 seconds of speech.
- Score <= 5: filler words > 8 OR grammar errors > 4 OR vocabulary score < 4.

**Re-test result:** The "um, yeah, I think it works" transcript now returns `fluencyScore: 2, grammarScore: 4`. The fix is a lookup table, not a prompt rephrasing.

## 17. Cognitive Engine Architecture Summary

To build a truly state-of-the-art developer platform like PinkyPoW and avoid the generic "AI wrapper" trap, we enforce the **PinkyPoW Cognitive Engine** and its stateful prompting methodologies. These principles elevate AI components from simple text-generators into highly deterministic, robust cognitive units.

### 17.1 The PinkyPoW Agentic Development Rules (System Guardrails)

When constructing the PinkyPoW platform, the AI assistant is governed by the following core rules to guarantee clean code, premium styling, and robust functionality:

1. **Decoupled Planning (Plan-Before-Code)**
   - Before executing code modifications, a complete research and planning phase must be conducted. This results in an `implementation_plan.md` artifact which defines exact file targets, additions, deletions, and verification steps. This guarantees that architectural context is preserved and prevents blind code modifications.
2. **Precision Modification (Targeted Block Diffing)**
   - The engine restricts file changes to line-specific block diffs rather than rewriting whole files. This prevents token waste, maintains unrelated codebase documentation/comments, and prevents formatting regression.
3. **Aesthetic-First Enforcement (Visual Premiumness)**
   - Every interface must feel premium, modern, and visually stunning. This includes:
     - Enforcing Outfit (for headings) and Inter (for body text) Google Fonts.
     - Curated color palettes with HSL variables instead of generic primary colors.
     - Interactive micro-animations (card hovers, pulse wave canvas audio visualizers, and glassmorphism).
     - Standardizing layout responsiveness.
4. **Isolated Execution Context**
   - The engine prevents directory hopping (never proposing `cd` commands) and keeps executions scoped strictly within the project directory workspace.
5. **Programmatic Assertions & Self-Healing**
   - Rather than assuming LLM compliance, the codebase enforces structural assertions (such as Zod validation, JSON parsers, and custom try/catch blocks).

---

### 17.2 Why Role Prompts Fail (The "Wrapper" Trap)

Standard role prompting (e.g., `"You are a Senior Software Engineer..."`) only achieves a fraction of the potential efficiency and reliability. The model is still prone to:

- Outputting Markdown fences (` ```json `) that break programmatic parsing.
- Hallucinating JSON keys (e.g., returning `"complexity"` instead of `"timeComplexity"`).
- Providing overly lenient or subjective grading (e.g., rating poor audio transcripts 9/10).
- Failing silently under network timeouts or rate limits.

To ensure high reliability and speed in production, PinkyPoW implements a multi-layered Cognitive Engine:

#### I. Dynamic Context-State Grounding

Every system prompt is injected with the user's real-time state fetched from MongoDB before execution.

- **Why it works:** The model doesn't operate in a vacuum. It knows the user's track (e.g., SWE, DevOps), their active streak, their current placement score, and their previous performance.
- **Example:** A DSA problem evaluation is dynamically tailored to the user's current track and skill tier, preventing the system from proposing irrelevant questions.

#### II. Negative Few-Shot Boundaries

Prompt templates include explicit negative examples (what *not* to output) alongside positive ones.

- **Why it works:** LLMs follow structured forms better when given strict anti-examples.
- **Prompt snippet:**

  ```text

  ```json
  { "result": { "timeComplexity": "O(n)" } }
  ```

  Instead, output ONLY raw JSON without markdown formatting or wrapping keys.

  ```

  ```

#### III. Lookup Rubrics over Generative Scoring

For grading tasks (speech pacing, vocabulary, and code complexity), generative ranges are replaced with strict tabular lookup rules.

- **Why it works:** Capping score ranges based on concrete conditions (e.g., "Score ≥ 9 requires: technical vocabulary present, 0 grammar errors, ≤ 2 filler words") forces deterministic scoring.

#### IV. Programmatic Self-Correction Loops

If `JSON.parse` or schema validation fails, the API route performs a single-shot retry containing the raw error message.

- **Why it works:** The model can inspect its own parsing failures (e.g., trailing commas, unescaped quotes) and fix them instantly without crashing the client UI.

## 19. Judges' Retrospective — Honest Assessment

### Where AI Actively Misled Us

**The markdown-wrapped JSON incident.** Documented in Section 6.1. The model was told not to use markdown fences. It used them anyway. The fix was a counter-example showing what a bad response looks like. **Lesson:** "Don't do X" is weaker than "here is what X looks like — never produce this."

**Lenient speech grading.** Documented in Section 6.2. The speech evaluator produced inflated scores because "grade on a scale of 1-10" with no rubric defaults to the model's training distribution, which skews generous. **Lesson:** For scoring tasks, the model needs a lookup table, not a range.

**Resume column merging.** Documented under P-20. `pdf-parse` and the model together failed on two-column resumes because the extraction layer produced garbled input. **Lesson:** AI output quality is bounded by input quality. Preprocessing is not optional.

**localStorage in edge middleware.** Documented under P-03. The AI assistant generated code that used `localStorage` inside Next.js middleware — an API that does not exist in the edge runtime. The code compiled without errors and the middleware silently passed all requests. **Lesson:** Type-checking is not enough. Architectural compatibility must be verified against the target execution environment.

**What we'd do differently.** Add JSON schema validation (using a library like Zod or Ajv) between `JSON.parse()` and the DB write on every AI route, not just the DSA one. Three of our remaining routes still rely on duck-typing the response — if the model decides to rename a field, those routes will silently store wrong data. That is the next fix.

### Where AI Accelerated Development

**Boilerplate speed.** Prompts P-01 through P-06 — global CSS, layout loading, font integration, DB connection helper, health endpoint, logging wrapper — were all generated in under two hours. Without AI assistance, that scaffolding would have taken most of day one. The saved time was invested in the DSA grader and speech evaluation — the features that differentiate the product.

**Cross-cutting concerns.** The Mongoose schema (P-07) was generated once with all required fields, indexing, and typing correct on the first attempt. No missing indexes, no missing timestamps, no duplicate model compilation errors. The AI had seen enough Mongoose boilerplate to get the details right immediately.

**Prompt self-correction.** The chain-of-thought guardrail in our system prompts (Section 4.5) was itself suggested by the model when asked to critique its own output after the markdown-fencing incident. That is a genuinely useful feedback loop — the model improving the prompts that control it.

**Consistent API design.** Every AI route follows the same structural pattern: validate input, query state, build prompt, call `generateJSON()`, validate output, mutate state, return response. This consistency was enforced by using the same AI prompt template (P-04) for every route. The result is 8 API routes that feel like they were written by one developer, not 8 different ones.

## 21. Key Engineering Decisions — Decision Log

| Decision                 | Options Considered                                  | Choice Made                   | Rationale                                                                                      |
| ------------------------ | --------------------------------------------------- | ----------------------------- | ---------------------------------------------------------------------------------------------- |
| Primary LLM provider     | OpenAI GPT-4, Groq Llama 3.3, Gemini 1.5 Flash      | Groq Llama 3.3 70B            | 3x faster inference. Cost-free for hackathon volume. JSON mode native support.                 |
| Failover strategy        | Single provider, round-robin, primary+failover      | Primary + failover            | Simplest reliable pattern. Groq handles 98% of calls; Gemini absorbs the remaining 2%.         |
| Temperature setting      | 0.0, 0.2, 0.5, 0.7                                  | 0.2                           | Low enough for deterministic JSON. High enough to avoid degenerate repetition in problem text. |
| Session storage          | localStorage, HTTP-only cookie, JWT in header       | HTTP-only cookie              | Only mechanism accessible in Next.js edge middleware. Prevents XSS token theft.                |
| DB connection strategy   | Per-request connect, connection pool, global cache  | Global cache (`global` var) | Prevents HMR connection leak. Single connection reused across all API routes.                  |
| Score persistence timing | Batch at end of session, per-action immediate       | Per-action immediate          | Ensures score is always current for the next AI prompt. No stale state risk.                   |
| AI output format         | Freeform text + regex parse, JSON mode at API level | JSON mode at API level        | Eliminates parsing failures. Model cannot produce non-JSON output.                             |
| Prompt rubric style      | Subjective range ("1-10"), mechanical lookup table  | Mechanical lookup table       | Produces consistent scores. Eliminates model's generous scoring bias.                          |
| Filter fallback strategy | Fail with error, return empty list, local keyword   | Local keyword fallback        | Application remains functional even without AI provider configured.                            |
